  setup, in a file unrelated to the diff, is a CI-runner infra flake — not a

## Lessons — this iteration (2)
- A well-written bug report CAN still be a small-fix first-mover candidate when the fix is
  genuinely small and non-design-sensitive: #65202 had an exact code-verifiable root cause
  (one gating condition) and a ~22-line fix with a mock-based fail-before test — took it same
  run even past DAILY_PR_HEALTHY, because that is the exact "strong backlog" exception, not
  volume for its own sake. Contrast #65194/#65195: reproduce-gate and design-surface filters
  correctly diverted those to a comment / bank.
- `_handle_topic_command`'s `not source.thread_id` guard for System-topic creation was the
  wrong signal — Telegram auto-creates a topic for the `/topic` message itself, so thread_id is
  present even on first activation. The correct "first activation" signal is reading topic-mode
  state BEFORE enabling it. General pattern: don't infer "is this the first time" from an
  incidental request attribute (thread_id) when the authoritative state (mode-already-enabled)
  is queryable.
